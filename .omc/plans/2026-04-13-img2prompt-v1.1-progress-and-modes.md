# img2prompt v1.1 — 分步进度显示 + 多模式视觉洞察工具箱

**Plan date:** 2026-04-13
**Scope:** Chrome MV3 extension (dist already shipping MVP v1.0)
**Branch target:** `main`
**Estimated effort:** 1–2 天（2 个主题并行推进）

---

## 1. Requirements Summary

用户在 v1.0 使用中提出两条核心反馈：

1. **识别过程不透明**：loading 只有转圈，看久了以为卡死。需要分步进度（截图 → 上传/压缩 → 模型识别 → 解析校验 → 完成），附耗时/状态文案。
2. **产品能力单一**：只输出绘图 prompt 太窄，想扩成"视觉洞察工具箱"：追加 **产品图片风格提取**、**网页设计风格提取** 等模式，共用截图/元素选择入口，按模式切换 system prompt 与输出结构。

目标：**不破坏 v1.0 架构**，追加"模式路由 + 进度事件流"两条横切能力。

## 2. Acceptance Criteria

所有 AC 必须可测、可观察：

- **AC-1** Loading 抽屉依次显示 5 个步骤徽标（截图 / 压缩上传 / 风格识别 / 结构解析 / 完成），当前步骤高亮，已完成步骤显示对勾与耗时（ms）。
- **AC-2** 每个步骤切换在 `<100ms` 内触达 UI（`chrome.tabs.sendMessage` PROGRESS 消息驱动，非轮询）；`LoadingMsg` 含单调递增 `seq: number`，side-panel 对 `seq` 降序消息直接丢弃，避免 fire-and-forget 导致的乱序。当前步骤附实时 `elapsedMs` 计数（每 120ms setInterval 刷新），取代固定 150ms 最小显示时长。
- **AC-3** 识别失败时，SW 必须先发 `LOADING_PROGRESS{step:'failed', failedAt:<prevStep>, errorCode}` **然后再** 发 `ERROR` 消息；side-panel 必须对两者任意到达顺序幂等（以 `seq` 与 `MSG.ERROR` 合并状态）。失败步骤红色标记，底部出现「重试」按钮（发送 `RPC_START_SELECTION`）。
- **AC-4** 启动抽屉 launcher 视图出现「模式切换」段控（≥ 3 个模式：`image_to_prompt` / `product_style` / `webpage_style`）；选中态通过 `chrome.storage.sync` 持久化，下次进入记住上次选择。
- **AC-5** 每个模式有独立的 `systemPrompt` 与 `zod` 输出 schema；`generateInsight<M extends Mode>(mode: M, ...)` TS 层仅接受声明模式；**跨进程边界**（`onMessage` 中的 `msg.mode`）必须经 `isMode(x): x is Mode` 运行时守卫，未通过则回落到 `DEFAULT_MODE` 并 `console.warn`（非崩溃）。
- **AC-6** 结果视图按 `mode` 路由到 3 个 Renderer（`PromptRenderer` 保持 v1.0 三卡 UI；`ProductStyleRenderer` 展示配色/材质/镜头/情绪标签；`WebDesignRenderer` 展示栅格/字体族/组件样式/交互动效）。
- **AC-7** 历史记录持久化新增 `mode` 字段；列表按模式图标区分；详情页进入对应 Renderer。旧数据（无 `mode`）默认视为 `image_to_prompt` 且可读。
- **AC-8** `SELECTION_COMPLETE` 消息携带 `mode`；handleSelectionComplete 根据 `mode` 分派到对应的 system prompt 与 schema。
- **AC-9** 所有新增 schema 必须通过 `vision-client.ts` 的 retry-once 机制（错误 system 消息带模式名）。
- **AC-10** 现有 40/40 测试全部通过；新增 ≥ 6 个测试：LoadingStepper 渲染、PROGRESS 消息派发、generateInsight 模式分派、每个模式 schema 的 zod parse（正/负样本）。

## 3. Implementation Steps

### Phase A — 进度事件流（~4h）

- **A1** `src/lib/messages.ts`：追加 `MSG.LOADING_PROGRESS`；扩展 `LoadingMsg` 为 `{ type, startedAt, step: LoadingStep, elapsedMs: number, mode?: Mode }`；新增 `type LoadingStep = 'captured' | 'cropped' | 'inferring' | 'parsing' | 'done' | 'failed'`。
- **A2** `src/background/index.ts` handleSelectionComplete：在 `captureP resolve`、`cropToBase64 resolve`、`generatePrompts` 调用前后、retry 时各 `sendToTab` 一条 `LOADING_PROGRESS`（带 elapsedMs = Date.now() - t0）。封装 `reportStep(tabId, step)` 辅助函数避免样板代码。
- **A3** `src/content/side-panel.tsx`：`View` 增加 `kind: 'loading'` 的子字段 `step` 与 `history: Array<{step,elapsedMs}>`；`incoming` 处理 `LOADING_PROGRESS` 时 append 到 history 并更新 current。
- **A4** 新建 `<LoadingStepper />` 子组件（在 `side-panel.tsx` 内定义，保持单文件紧凑）：5 个步骤纵向 timeline + 圆点 + 进度文字；当前步骤呼吸动画；完成步骤显示 `(320ms)` 耗时。样式添加到 `side-panel-styles.ts`（`.stepper-*` 类）。
- **A5** 错误态：`MSG.ERROR` 到达时，最后一个进行中步骤标红，底部渲染「重试」按钮 → 点击发送 `RPC_START_SELECTION`。

### Phase B — 模式路由（~5h）

- **B1** 新建 `src/lib/modes.ts`：导出 `type Mode = 'image_to_prompt' | 'product_style' | 'webpage_style'`；`MODE_META: Record<Mode, {label, icon, description}>`；`DEFAULT_MODE`。
- **B2** 重构 `src/lib/system-prompts.ts`：`buildSystemPrompt(mode, language)` 多分支返回；保留原 `image_to_prompt` 分支文案不变。
- **B3** 新建 schemas：
  - `src/lib/schemas/prompt-schema.ts` ← 现有 `prompt-schema.ts` 重命名移入。
  - `src/lib/schemas/product-style-schema.ts`：`{ palette: string[]; materials: string[]; lighting: string; mood: string[]; camera: string; tags: string[] }`
  - `src/lib/schemas/webpage-style-schema.ts`：`{ layout: string; typography: {heading, body}; colors: {primary, accents}; components: string[]; interactions: string[]; tone: string }`
  - `src/lib/schemas/index.ts`：导出 `SchemaByMode: Record<Mode, ZodType>` 与 `InsightByMode: { image_to_prompt: PromptResult; product_style: ProductStyle; webpage_style: WebDesign }`。
- **B4** 重构 `src/lib/vision-client.ts`：`generatePrompts` 改名 `generateInsight<M extends Mode>(mode: M, opts) → Promise<InsightByMode[M]>`；内部按 `mode` 取 system prompt + schema。老的导出名保留 deprecated alias 供 background 调用切换平滑。
- **B5** `src/lib/messages.ts`：`SelectionCompleteMsg` 与 `ResultPayload` 增加 `mode: Mode`；`HistoryEntryDto.prompts` 字段重命名为 `insight: InsightByMode[Mode]`（旧 `prompts` 字段读侧兼容：迁移时 `prompts ? {mode:'image_to_prompt', insight: prompts} : entry`）。
- **B6** `src/background/index.ts`：`handleSelectionComplete(msg, tabId)` 读取 `msg.mode`；调用 `generateInsight(msg.mode, ...)`；`storage.addAndPrune` 写入 `{ mode, insight }`。
- **B7** `src/content/overlay.tsx`：`SelectionCompleteMsg` 构造时从 `launcher` 选中的 `mode` 传入（mode 从 content script 层的 module 变量读取，由 side-panel 通过自定义 DOM event 或共享 state 写入）。
- **B8** `src/content/side-panel.tsx` launcher：在三卡片上方增加 `<ModeSegment />`，读写 `chrome.storage.sync.mode`；选中值通过 `window.postMessage` 或共享模块变量暴露给 overlay，发起截图时附上。
- **B9** 结果渲染分派：`<ResultBody payload={payload} />` 内部 `switch(payload.mode)` 到三个 Renderer 子组件；`PromptRenderer` 即原有 UI 搬迁；新两个 Renderer 采用标签 chip + 色块 + 列表 3 分区布局。
- **B10** 历史列表：`<HistoryGrid />` 每条卡片左上角加 `MODE_META[mode].icon`；详情页进入对应 Renderer。

## 5. RALPLAN-DR Summary

### Principles

1. **不破 v1.0**：MVP 用户路径（截图→转圈→3 张 prompt）零回归，历史数据必须可读。
2. **进度 UI 由事件驱动**：service worker 是真实时间源，UI 只渲染收到的 PROGRESS 事件，避免前端假动画欺骗用户。
3. **模式是横切维度**：`Mode` 贯穿 launcher → SELECTION_COMPLETE → handleSelectionComplete → vision-client → result renderer → history；单点改动可扩新模式。
4. **Schema 即合约**：每个模式的输出结构用 zod 声明，一处失败 retry 一次，失败即红色步骤 + 重试。

### Decision Drivers

1. **交互可信度 > 速度**：用户更怕"不确定卡死"而非等 2 秒。
2. **扩展成本 < 单次开发成本**：新增第 4 个模式应只要 1 个 schema + 1 个 renderer。
3. **保留 MVP 代码不做大重构**：在现有文件结构上"横切"而非"重写"。

### Viable Options

#### Option A（推荐）— 事件驱动 PROGRESS + Mode 路由表

- **Pros**：侵入面小；UI 真实反映后端进度；新模式 = schema+renderer 两处改动；测试易写。
- **Cons**：SW→content 要多发 4–5 条消息，极端卡顿网络下仍可能步骤跳变。

#### Option B — 前端定时推进的假进度条

- **Pros**：改动更小，无需 SW 改造。
- **Cons**：会在真实卡死时继续"走进度条"，放大用户误判；违背 Principle 2。**Rejected.**

#### Option C — SSE/流式响应（真实 token 级进度）

- **Pros**：最真实的推理进度。
- **Cons**：OpenAI 兼容 API 的 `response_format: json_object` 与流式难以同时稳定；单次 <3s 的短响应，流式复杂度 ROI 不划算。**Rejected for v1.1, revisit if latency > 8s.**

## 6. Risks & Mitigations

| # | 风险 | 影响 | 缓解 |
|---|---|---|---|
| R1 | content script 与 overlay 共享当前 mode 时发生跨 iframe / 重新加载导致 mode 丢失 | 截图发起用了错误模式 | Overlay 直接在 `startSelection` 消息 payload 读 `mode`；content 层在 START_SELECTION 消息里从 storage.sync 取最新值后再挂载 overlay。 |
| R2 | 历史数据迁移：旧记录无 `mode` 字段 | 历史页渲染崩溃 | `storage.list()` map 时兜底 `mode ?? 'image_to_prompt'`，`insight ?? {prompts: entry.prompts}`。 |

## 8. ADR

- **Decision**：以 **Option A（事件驱动 PROGRESS + Mode 路由表）** 实现 v1.1。
- **Drivers**：交互可信度 > 速度；扩展成本 < 单次开发成本；最小侵入。
- **Alternatives considered**：B（前端假进度条）— 违背信任原则，拒绝；C（SSE 流式）— ROI 不划算，延后到 v2 若推理时延 > 8s 再考虑。
- **Why chosen**：让 UI 真实反映 SW 真实进度；Mode 作为贯穿维度，使 v2/v3 加模式只需新增一个 schema + renderer；零破坏 v1.0 数据。
- **Consequences**：+1 横切 Mode 类型、+4 条消息往返、+3 个 zod schema；history 表结构新增字段但向后兼容。
- **Follow-ups**：v1.2 预留 "UI 配色提取" "字体识别" 两个 mode 位；v2 评估 SSE；Tauri 桌面端复用 `src/lib/{modes,schemas,vision-client}` 整块。

## 8a. Iteration-2 Delta (binding — overrides earlier sections where conflict)

### Messages & contracts (override §2 AC-2/3 + §3 A1)

```ts
// src/lib/messages.ts
export type LoadingStep = 'captured' | 'cropped' | 'inferring' | 'parsing' | 'done' | 'failed';
export interface LoadingMsg {
  type: typeof MSG.LOADING;     // renamed: MSG.LOADING_PROGRESS collapsed into LOADING with step field
  seq: number;                  // monotonic per selection; side-panel drops out-of-order
  startedAt: number;
  step: LoadingStep;
  elapsedMs: number;
  failedAt?: LoadingStep;       // set only when step === 'failed'
  errorCode?: ErrorCode;        // set only when step === 'failed'
  mode: Mode;
}
export interface HistoryEntryDto {
  id: string; createdAt: number; pageUrl: string;
  mode: Mode;                                 // new required field
  insight: InsightByMode[Mode];               // discriminated union; replaces `prompts`
}
```

Read-side compat for legacy `prompts` rows handled in `storage.list()` adapter (see §8a Storage).

### Storage migration (resolves Architect #3, #4)

- `storage.ts` bump `DB_VERSION` from 1 → 2. `onupgradeneeded(v:1→2)` is a **no-op schema change** that only forces a version transaction; existing rows untouched.
- Add `normalizeRow(raw): HistoryEntry` in `storage.ts`:
  ```ts
  function normalizeRow(raw: any): HistoryEntry {
    if (raw.insight && raw.mode) return raw;
    // v1.0 row: { prompts: PromptResult }
    return { ...raw, mode: 'image_to_prompt', insight: raw.prompts };
  }
  ```
  `list()` / `getOne()` pipe all rows through `normalizeRow`.
- New writes always use `{ mode, insight }`; never write legacy shape again.
- **Test C7**: write a legacy-shape row directly via IDB → `list()` returns normalized shape.

### Mode propagation to overlay (resolves Architect #6; overrides §3 B7/B8)

- Launcher writes `chrome.storage.sync.set({ mode })` on change.
- `content/index.ts` `START_SELECTION` handler becomes **async**: `const { mode } = await chrome.storage.sync.get('mode'); mountOverlay({ mode: mode ?? DEFAULT_MODE });`.
- `mountOverlay({mode})` passes `mode` as constructor prop to the overlay React root; overlay's `SelectionCompleteMsg` constructor reads `props.mode` (immutable once mounted).
- No shared module variable, no postMessage. Single source of truth = `chrome.storage.sync`.
- **Test C8**: unit-level mock `storage.sync.get('mode')` → assert overlay receives expected mode in its `SELECTION_COMPLETE` payload.

### Runtime mode guard (resolves #8; overrides §3 B6)

```ts
// src/lib/modes.ts
export const MODES = ['image_to_prompt','product_style','webpage_style'] as const;
export type Mode = typeof MODES[number];
export const DEFAULT_MODE: Mode = 'image_to_prompt';
export function isMode(x: unknown): x is Mode {
  return typeof x === 'string' && (MODES as readonly string[]).includes(x);
}
```

`handleSelectionComplete` opens with `const mode: Mode = isMode(msg.mode) ? msg.mode : (console.warn('[img2prompt] bad mode', msg.mode), DEFAULT_MODE);`.

### `buildSystemPrompt` object-arg (#5; overrides §3 B2)

```ts
// src/lib/system-prompts.ts
export function buildSystemPrompt(args: { mode: Mode; language: Language }): string { ... }
```

All callers must switch to object-arg. Positional legacy signature deleted, not aliased — fail fast in TS.

### Renderer file split (#10; overrides §3 A4/B9)

- `<LoadingStepper />` moves to `src/content/loading-stepper.tsx`.
- Renderers move to `src/content/renderers/{prompt,product-style,webpage-style}.tsx`.
- `side-panel.tsx` keeps only the view-state-machine + shell + launcher; imports all 4 components. Target file budget: `side-panel.tsx` ≤ 500 LOC post-refactor.

### PROGRESS event `seq` + await-discipline (#7)

- SW-side helper `async function reportStep(tabId, mode, step, t0, seqRef, extras?)` increments `seqRef.current` then `await sendToTab(...)`. All PROGRESS sends go through this helper.
- Even if `await` slips, `seq` guarantees side-panel can drop out-of-order messages. Belt & suspenders.

### Removed

- §6 R4 "150ms UI 节流" is **removed**; replaced by live `elapsedMs` counter on the current step (driven by `setInterval(120ms)` clearing on next PROGRESS).
